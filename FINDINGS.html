What this changes

Founder/CEO still converts the best — but by roughly 3×, not 5×+. CMOs convert at 5.2% with a tight confidence interval. That's worse than Founder, but not "essentially zero" as the original framing implied.

The original "1 in 15" framing was based on five titled CMO replies in the v3 prompt-version subset. With five rows, the 95% CI on a 1/5 sample is [3.6%, 62.4%] — too wide to support a categorical drop.

3The LLM is the sole agency filter today

Every accepted/rejected row in prospeo_new_leads has filter_method=llm. The cheap deterministic rule pass (agency tokens, marketplace blocklist) caught zero of the 35 leads. SQL-verified by inspecting the table directly.

Why this matters for Victor's category test

ARCHITECTURE.html §4 frames the filter as "rules first, then LLM for borderline cases." In practice, every lead is a borderline case in current data — the rules are too narrow to catch anything. So when we run the category-mode pilot, brand-rate will be governed entirely by how Haiku judges the larger volume of marginally-relevant candidates that categories pull in. The pilot needs to measure the LLM's behavior, not just the final accept-rate.

Spot-checking the 14 rejected leads turned up one defensible mis-rejection and one correctly rejected lead. Pending pilot for a full audit:

  • BANDO — rejected as agency. API-verified: both ban.do (DTC lifestyle brand) and Bando Group (Japanese industrial-belt manufacturer) are real product brands. This is a defensible mis-rejection.
  • Bionic Prosthetics & Orthotics — rejected as agency. API-verified: actually a clinical-services chain (12 states) that fits prosthetics for patients, not a product brand. The LLM was correct here.

The pilot will log Haiku's full reasoning for every accept/reject so Jam can audit at scale.

5Bug fix — SSL drop on startup

Bench-verified reproduction:

File "prospeo_sync.py", line 253, in fetch_domains_with_decision_maker
    cur.execute(sql)
psycopg2.OperationalError: SSL error: unexpected eof while reading

The query scans all 230k rows of lead_contacts for any with a chief/owner/marketing title. On some networks the Supabase pooler closes the SSL connection mid-query and the bot aborts before doing real work.

Fix shipped

  • db.py:43-65 — added statement_timeout=300000 (5 min), TCP keepalives, and connect_timeout=10.
  • prospeo_sync.py:243-285 — wrapped fetch_domains_with_decision_maker in a 3-attempt retry that opens a fresh connection on transient drops. Caller's connection left untouched.

Verification: python run.py scrape-leads --dry-run --limit 50 now completes. Was failing every attempt before.

6The category-pilot script — what was verified, what wasn't

Before pushing the pilot live, multiple Prospeo filter shapes and value formats were probed empirically. Findings:

Filter shape — empirically resolved

Shape triedResult
filters.company_industry.include = [...] with verified value strings Works — credits charged, real results returned
filters.company.industries.include (nested, plural) Silently ignored — accepted as syntax but total_count identical to no-industry baseline
filters.company.industry.include (nested, singular) Silently ignored — same pattern
filters.company_industries.include (top-level, plural) INVALID_FILTERS
filters.industries.include / filters.industry.include INVALID_FILTERS

Bug almost shipped

The initial research said the shape was filters.company_industry.include with strings like "Apparel and Fashion" (LinkedIn's pre-2023 taxonomy name). That string is not in Prospeo's enum and returns INVALID_FILTERS. If the pilot had been run as originally written, the entire Pass B would have 400'd. API-verified by checking 27 candidate industry strings: 14 accepted, 13 rejected.

Verified industry list now used by the pilot

Retail Apparel and Fashion Apparel Manufacturing Cosmetics Personal Care Product Manufacturing Food and Beverage Manufacturing Furniture and Home Furnishings Manufacturing Sporting Goods Manufacturing Consumer Goods Pet Services

Each value was tested live and confirmed to (a) parse cleanly and (b) actually narrow Prospeo's result count. Strings that look correct but are rejected and would have crashed the pilot: "Apparel and Fashion", "Apparel", "Retail", "Furniture", "Manufacturing", "Fashion", "Apparel & Fashion" (ampersand), "Personal Care Products", "Health and Beauty", "Food Production", "Beverage Manufacturing", "Retail Home Furnishings", "Retail Consumer Goods", "Health, Wellness & Fitness".

What's still unverified

  • Pending pilot company_location_search with raw strings like "United States". Kept behind --with-country flag (off by default) until validated. The earlier research suggested it requires Suggestions-API-resolved values — but we never found a working call to the Suggestions API.
  • Pending pilot Actual brand-rate Pass B will achieve after the LLM filter. This is the whole point of the pilot.
  • From repo docs The 15-domain cap on company.websites is trusted from the team's documented 506-credit incident; not re-tested.

How to run it

The script has been re-verified by dry-run after every change. Safety guards:

  • Hard budget cap (--max-credits) checked before every Prospeo call.
  • 15-domain batching for Pass A — protects against the silent-ignore bug.
  • Runaway-result safety check — abort batch if total_count > 75.
  • No DB writes. Pilot only reads CSV and writes XLSX. prospeo_new_leads untouched.
  • Pause-and-confirm [y/N] prompt before any live call.
  • Per-pass LLM reasoning logged to the output XLSX for audit.

Decision 3 — Title-enrichment backfill (separate ticket)

649 of our 1,418 positive-family leads have no title in lead_contacts (46% of all positives). They sit in a 3,673-lead untitled pool. The repo has a name_extraction.py + backfill_names_from_replies.py pipeline that already runs Haiku over email-signature bodies for first/last-name backfill (Option 4 architecture in BACKFILL_NAMES_FROM_REPLIES_PLAN.md). It does not currently extract job titles — but the LLM-extractor stage has the signature context to do so with a prompt change. A parallel "titles" prompt + DB column would slot in alongside the existing flow. Out of scope for the scraper, but worth flagging for whoever owns the classifier side.
