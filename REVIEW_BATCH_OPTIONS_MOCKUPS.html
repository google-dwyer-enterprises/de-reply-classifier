Review-batch scoping — UI mockups

Three structural fixes for the cross-batch contamination risk. Each removes the "Jam must remember to filter" problem in a different way. Side-by-side mockups + a comparison table at the end.

Option 1 Per-batch view (worker auto-creates)

When the worker marks a batch ready, it also creates a new NocoDB grid view named "Review batch #N" with a hardcoded filter on this batch's id. The email button opens THAT view directly. Jam cannot see other batches from inside it.

What it does

Worker calls NocoDB API at mark_ready time:

  1. POST /views with filter scrape_request_id=15
  2. Enable public share, get UUID
  3. Build email URL with that UUID

"Select all" is now genuinely scoped to the batch. Jam can't bulk-approve cross-batch even if she tries.
Tradeoffs   ✅ Strongest isolation. Zero filter trust required.   ❌ Worker code change in notifier.py + worker.py. NocoDB token becomes a Railway env var. Cleanup logic needed (delete view on finalize). ~45 min + PR.

Option 2 — Recommended Linked records inside expanded parent

Set up a NocoDB "Has Many" link from scrape_requests → prospeo_new_leads (one-time config; backed by the existing FK). Jam clicks her batch's row in All requests, expands it, and a Linked Records panel shows ONLY that batch's leads — editable inline.

Step 1 — Email → opens her scrape_requests row (already)

Lead batch #15 ready for review — 3 leads scraped
Hi Jam,
Batch #15 finished scraping.

Open request #15 in NocoDB →

Same email button you already have. Opens her parent row.

What changes

Inside scrape_requests table, add one new column:

  • Type: Links (Has Many)
  • Linked Table: prospeo_new_leads
  • Linked Field: scrape_request_id

NocoDB auto-populates the link from the existing FK data. No DB migration.

Why it's safe

The Linked Records panel is structurally bound to row #15 by NocoDB. It can never show leads from another batch — they're not linked to this row.

No header checkbox at the panel level (NocoDB shows links one-at-a-time); for bulk, you scroll the list and click each chip.
Inline-edit UX depends on NocoDB version. Some versions show the linked-records panel as read-only initially; clicking a row opens an inline editor. Needs a 10-min verify on your NocoDB.
Tradeoffs   ✅ Cheapest to implement (~10 min API setup, zero worker code change). Native NocoDB UI. Cleanup is automatic.   ❌ NocoDB version may limit inline edits. Bulk-edit inside a linked-records panel is one row at a time (no header-checkbox in the panel).

Option 3 Button column → opens a pre-filtered view

Add a NocoDB "Button" column on scrape_requests. Each row's button opens the Review batch view with a URL param like ?filter=scrape_request_id:15. NocoDB respects the URL filter, so the view shows only that batch's leads. The button replaces the email's Open-in-NocoDB link.

Why it's safer than today

Jam can't reach the unfiltered Review batch view from her normal flow — the only entry points (email link, Review button) carry the per-batch URL param.

If Jam navigates directly to the Review batch view (bookmark, manual click), the URL has no filter param and she sees all pending leads again. The protection is conditional on the entry point, not absolute.
Tradeoffs   ✅ Same Review batch view (no duplicate views); URL param is the scoping. Adds a per-row Review button which is convenient navigation.   ❌ Protection isn't absolute — direct navigation bypasses it. URL filter syntax has to be tested in your NocoDB version.
