Prospeo lead scraper

Pulls decision-maker leads from Prospeo for domains in our inclusion list, filters out agencies/resellers, and writes accepted brand contacts to prospeo_new_leads + a CSV for Jam to load into Instantly.

How it works

inclusion CSV (Website column) │ load_domains_from_csv() — auto-detects CSV/TSV, normalizes URLs ▼ domains list (~52k after cleaning) │ dedupe vs lead_contacts/replies/prospeo_new_leads │ skip domains where we already have a decision-maker ▼ batches of 50 domains │ POST /search-person (1 credit per 25-result page) │ filters: company.websites + person_job_title (Smart match) ▼ person_ids of decision-makers │ POST /enrich-person (1 credit per verified email found; free otherwise) │ only_verified_email: true — avoids paying for unverified emails ▼ leads with email + company info │ rule filter (agency tokens / marketplace) → reject │ LLM grey-zone (Haiku, prompts/agency_filter.txt) → brand/agency/reseller ▼ prospeo_new_leads (Supabase) + exports/prospeo_new_leads_<timestamp>.csv

Setup

  1. Add to .env: 
    PROSPEO_API_KEY=<your key>
  2. Run the new tables once in Supabase (see migrations.sql bottom):
    • prospeo_new_leads
    • domain_inclusion_list (optional — only if loading domains from DB)
  3. Clean the inclusion list. Drops .org / .edu / .io / gmail / yahoo / outlook AND ~80 mega-brand domains (Adidas, Walmart, Pfizer, etc. — they're not real targets and their massive exec rosters break Prospeo's batched filter): 
    python scripts/clean_inclusion.py original_data/inclusion.csv original_data/inclusion_clean_v2.csv
    The script names suffix _v2 in current use — bump when MEGA_BRANDS or BLOCKED_* lists change.

CLI

scrape-leads — main scraper

# Sanity check, no API calls, no DB writes
python run.py scrape-leads --domains original_data/inclusion_clean_v2.csv --dry-run --limit 50

# Small live test (~$0.04 on GROWTH plan)
python run.py scrape-leads --domains original_data/inclusion_clean_v2.csv --limit 50

# Safe weekly drop with a hard budget cap (recommended)
python run.py scrape-leads --domains original_data/inclusion_clean_v2.csv --limit 1000 --max-credits 80

# Skip the LLM grey-zone classifier (rule-based filter only)
python run.py scrape-leads --domains original_data/inclusion_clean_v2.csv --limit 500 --skip-llm

# Email + mobile in a single run (10 credits per mobile, accepted leads only)
python run.py scrape-leads --domains original_data/inclusion_clean_v2.csv --limit 500 --with-mobile

# Full run from the cleaned CSV
python run.py scrape-leads --domains original_data/inclusion_clean_v2.csv

Flags:

  • --domains <csv> — input file. Accepts CSV or TSV; auto-picks domain / Website / URL column, or falls back to the first column.
  • --limit N — cap domains processed (after dedupe).
  • --dry-run — load + dedupe only, exit before API calls.
  • --skip-llm — skip Haiku grey-zone agency classifier.
  • --with-mobile — after filtering, also enrich mobile numbers for accepted leads only (10 credits per verified mobile; free if not available).
  • --max-credits N — hard budget cap. Aborts the run before any API call that would push the credit count past N. Defaults to no cap. Strongly recommended on every run after the 506-credit burn we had with the old batch size.
Live progress: each batch prints a status line like [batch 7/67] credits: 23/80 | accepted: 4 | rejected: 6 so you can monitor spend in real time. Leads are written to Supabase per batch — if the script crashes or is killed mid-run, everything found up to that point is safely persisted, and the next run resumes automatically by skipping already-scraped domains.

enrich-mobile — catch-up command

Adds mobile numbers to accepted leads in prospeo_new_leads that don't have one yet. Use this when you scraped email-only and decided later to add mobiles (e.g. after Jam reviews the accepted list).

# Show how many leads would be enriched + estimated cost. No API calls.
python run.py enrich-mobile --dry-run

# Enrich every accepted-but-no-mobile lead in the DB
python run.py enrich-mobile

# Cap to N leads (useful for budget-bounded catch-ups)
python run.py enrich-mobile --limit 50

Selects where rejected = false and mobile is null. Re-runnable safely — already-enriched rows are excluded automatically.

export-leads — cumulative dump from DB

Per-run CSV/XLSX files only contain that run's results. Use this to dump the entire prospeo_new_leads table at any time. No API calls, no cost.

# Generate a fresh cumulative file pair
python run.py export-leads

Writes exports/cumulative_<timestamp>.csv (accepted brands only) and exports/cumulative_<timestamp>.xlsx (Accepted + Rejected sheets, with filter reasoning). Run after each scrape if you want a single up-to-date file to hand to Jam.

Cost model

GROWTH plan, $99/mo for 5,000 credits ≈ $0.02/credit.

  • search-person: 1 credit per page of 25 results returned (paginated).
  • enrich-person (email only): 1 credit only if a verified email is returned, free otherwise.
  • enrich-person (mobile, opt-in via --with-mobile or enrich-mobile): 10 credits per verified mobile, free if not available. Coverage typically 30–50% of accepted leads.
  • Haiku grey-zone: ~$0.0001 per lead classified.

Observed on a 47-domain pilot with the owner-only title list:

  • ~2 Prospeo credits → ~$0.04
  • 1 verified email out of 5 decision-makers found.

Projected for the full 52k inclusion list (email only):

  • ~500–1,500 credits ($10–$30)
  • ~500–1,500 verified owner-level leads.

If mobile is enriched for every accepted lead at that scale (worst case all have a verified mobile):

  • Up to ~5,000–15,000 additional credits ($100–$300) — only charged for mobiles actually returned.
  • Realistic at ~30–50% coverage: ~$30–$150 extra.
Budget note: GROWTH gives 5,000 credits/month. Email-only full inclusion list fits in one month's quota; weekly drops of ~5k domains fit comfortably. Adding mobile across the full list does NOT fit in one month — schedule mobile enrichment selectively (e.g. only on leads Atlas will actually cold-call) or spread across multiple monthly quotas.

Filter design (current)

Job titles — owner-only

PROSPEO_TITLES = [
  "CEO", "Chief Executive Officer", "Founder", "Owner", "Co-Owner",
  "President", "Founder and CEO", "Managing Director", "Chairman",
]

This list was derived from Dwyer's own reply data (classifications v3, 166 classified replies with a matched lead title). Owner-level titles produce essentially all positive outcomes; marketing/CMO/e-com-manager titles produced 1 booking out of 15 replies in our history. The previous broader list is kept as _PROSPEO_TITLES_BROAD_REF in prospeo_sync.py for re-tuning runs.

Agency / reseller / marketplace filter

  • Rule pass (cheap, deterministic):
    • Agency tokens in company name/website: agency, marketing, consulting, consultancy, services, solutions, media, digital, growth, fulfillment, 3pl, partners, advisors, advisory
    • Known marketplaces: amazon.com, walmart.com, etsy.com, ebay.com, faire.com, shopify.com, alibaba.com, aliexpress.com
  • LLM pass (Haiku, prompts/agency_filter.txt) — runs only on leads the rule pass couldn't decide. Classifies brand / agency / reseller / marketplace / unknown with a one-line reason.

Only brand lands in the CSV export. The other categories stay in prospeo_new_leads for audit but with rejected = true.

Re-tuning the title list

The title list is a snapshot of what's working today. Re-run this query monthly against fresh classifications data and adjust:

with latest as (
  select distinct on (reply_id) reply_id, lead_email, label
  from classifications where prompt_version='v3'
  order by reply_id, classified_at desc
)
select lower(trim(lc.title)) as title,
       sum(case when l.label in ('booked','interested','interested_past')
                then 1 else 0 end) as positive,
       sum(case when l.label in ('not_interested','not_now')
                then 1 else 0 end) as negative,
       count(*) as total
from latest l
join lead_contacts lc on lc.lead_email = l.lead_email
where coalesce(lc.title,'') != ''
group by title
order by positive desc, total desc
limit 30;

Drop titles with 0 positives across 5+ replies. Promote titles that show consistent positive signal (≥3 positives, ≥30% positive rate).

Things that have bitten before

  • /domain-search is deprecated (removed March 1, 2026). Use /search-person + /enrich-person two-call workflow.
  • Websites filter has an undocumented 15-domain limit. The docs say 500, but empirically the filter is silently ignored above 15 entries — Prospeo returns unfiltered global results and charges for every page. We burned 506 credits ($10) discovering this with batch size 50. PROSPEO_BATCH_DOMAINS is now hard-coded to 15, and a safety check aborts any batch returning more than 15 × 5 = 75 results.
  • Industry filter shape is filters.company_industry.include (singular, top-level) — NOT nested under company like websites is. Common wrong shapes (empirically verified 2026-05-14, scripts/verify_prospeo_shape.py):
    • filters.company.industries.include — accepted as syntax but silently ignored (returns the no-industry baseline total_count). Same class of bug as the 15-domain websites case.
    • filters.company.industry.include — same silent-ignore.
    • filters.company_industries.include (plural) — 400 INVALID_FILTERS.
    • filters.industries.include / filters.industry.include (top-level) — 400 INVALID_FILTERS.
  • Industry values use Prospeo's post-2023 LinkedIn taxonomy. The pre-2023 names are silently rejected. Example: "Apparel and Fashion" (old) → INVALID_FILTERS: Invalid industry(s) in include list; use "Retail Apparel and Fashion" (current). Verified e-com strings: Retail Apparel and Fashion, Apparel Manufacturing, Cosmetics, Personal Care Product Manufacturing, Food and Beverage Manufacturing, Furniture and Home Furnishings Manufacturing, Sporting Goods Manufacturing, Consumer Goods, Pet Services. See FINDINGS.html §6 for the full list and rejections.
  • 400/INVALID_FILTERS body shape: the specific failure reason is in filter_error, not error_toast or message. Log filter_error first when debugging shape problems.
  • Mega-brands break batched searches. A single Fortune-500 in a 15-domain batch (Adidas, ADM, Pfizer…) brings in 100+ exec-titled rows. The cleaner now drops ~80 known mega-brand domains via MEGA_BRANDS in scripts/clean_inclusion.py. Extend the list as new ones surface.
  • Filter shape — person filters: person filters are top-level keys (person_job_title, person_seniority) — NOT nested under a person object. Wrong nesting → 400 INVALID_FILTERS.
  • Empty result sets come back as 400 NO_RESULTS, not 200 with an empty array. _search_people treats 400/NO_RESULTS as success with zero hits (no credits charged).
  • only_verified_email: true causes Prospeo to return 400 / NO_MATCH when no verified email exists for a person. We treat that as a soft skip, not an error. Without this flag we'd pay credits on unverified emails that bounce and hurt sender reputation.
  • Subdomains are rejected. parts.agcocorp.com → 400 INVALID_FILTERS: Subdomains are not supported. _norm_domain strips subdomains down to the registrable domain. Multi-part TLDs (e.g. brand.com.au, brand.bc.ca) are preserved via _MULTI_PART_TLDS. Extend that set when Prospeo rejects a previously-fine domain.
  • Person IDs can stale — if a Search result sits unused for too long, the matching enrich-person call returns NO_MATCH. Don't persist person_ids across days; enrich immediately after search.
  • LLM responses sometimes ship inside ```json fences even when prompted not to. The parser strips fences before json.loads. If the prompt model changes, re-verify.
  • Seniority filter is too broad. Filtering by Vice President returns every VP — Engineering, Finance, Operations. Use person_job_title with Smart match for precision.
  • SSL drops on fetch_domains_with_decision_maker. The 230k-row scan over lead_contacts occasionally drops the Supabase pooler connection mid-query. The function is wrapped in a 3-attempt retry with a fresh connection per retry; if you refactor it, preserve the retry. Also: db.connect() sets statement_timeout=300000 and TCP keepalives to keep the connection warm during long scans.
  • Always run with --max-credits. The script also has internal safety checks, but a hard budget cap is the only thing that's guaranteed to stop a runaway pathological case.
