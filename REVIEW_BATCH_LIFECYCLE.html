Per-batch review view — full lifecycle

From form submission to view cleanup. Each phase shows: scrape_requests row on the left, per-lead state in prospeo_new_leads in the middle, NocoDB sidebar on the right. Cleanup triggers are highlighted in yellow. Worked example: batch #15, 3 leads (Fashion Nova, North Face, DBO Labs).

Cleanup rule: the per-batch view gets deleted when the batch leaves the "needs review" state via ANY path:
  • status auto-flips to moved (every lead decided, all approved moved — finalize sweep)
  • approval set to approved (Jam mass-approves)
  • approval set to rejected (Jam mass-rejects)
All three triggers are independent and idempotent — view deletion checks "if view_id exists, delete it" so multiple triggers can't double-delete.

Summary: what gets deleted vs preserved

ObjectDeleted on cleanup?Why
Per-batch NocoDB view YES — view + its public share are destroyed via API. Row's review_view_id + review_url cleared. Sidebar stays clean. View was only useful while the batch needed review.
scrape_requests row Never. Stays at status='moved' in the All requests grid. Audit trail of "what was requested + scraped + cost in credits". Permanent record.
prospeo_new_leads rows Never. All 3 stay with their final lead_approval values. Audit trail of "what BC returned + what Jam decided per lead". Permanent.
lead_contacts entries Never. Approved leads stay in the 200K pool forever. That's the whole point — landing leads in the pool.
Email Jam received Survives — the URL inside still resolves to the deleted view, which now 404s. Acceptable: she's done with the batch. The All requests grid is the canonical record.
