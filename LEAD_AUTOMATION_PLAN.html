Lead Scrape Automation — Implementation Plan

Jam-driven NocoDB form → Railway worker → BetterContact scrape → email → per-lead approve/reject → 200K pool. Shipped on main at 102f0a4 (PR #4); per-lead workflow added on follow-up branch.

What this document is

The end-to-end implementation plan for the Lead Scrape Automation feature: what we built, why each piece exists the way it does, and what was verified before merge. Deployment steps (Railway dashboard clicks, env-var values) live in LEAD_AUTOMATION.md.

1Why this exists

Jam can't trigger lead scrapes herself today. She pings the engineer, who runs python run.py scrape-leads ... from a Windows machine, hands over a CSV, and Jam manually copies leads into lead_contacts (the 230,227-row "200K pool"). Every batch adds engineer-time friction and gates Jam on Anna's availability.

The outcome we wanted: Jam opens NocoDB, fills a one-line form ("I want 500 leads"), and the rest happens without an engineer. A Railway worker picks up the request, runs the BetterContact scraper, emails Jam when results are ready, and she reviews each lead in NocoDB — approve/reject per row. As leads get approved, the worker keeps moving them into lead_contacts; she can come back over multiple sessions, and the batch auto-finalizes when every lead has a decision.

3State machine

Request-level: scrape_requests.status

statusset bymeaning
pendingJam (NocoDB form insert)Waiting for the worker to pick it up.
runningworker (claim_pending_request)Scrape in flight. Tracked with started_at for the stuck-running sweep.
readyworker (mark_ready)Scrape finished, leads in prospeo_new_leads. Email sent. Jam reviews per-lead inside NocoDB.
movedworker (finalize_request_if_done)Every lead has a decision (no lead_approval='pending' left) AND every approved lead has been moved. Terminal.
failedworker (any exception path)BC error, DB error, or move-mapping mismatch. error_message populated. Terminal until manual intervention.

Lead-level: prospeo_new_leads.lead_approval + lead_moved_at

The per-lead workflow (Flavor C / granular). NULL on rows from the CLI scraper; for worker-tagged rows, seeded based on BC's filter outcome and then driven by Jam's decisions:

lead_approvallead_moved_atmeaning
pendingNULLBC accepted the lead; Jam hasn't decided yet. Worker keeps the batch in status='ready' while any leads are here.
approvedNULLJam approved; queued for the next worker move cycle (within ~60s).
approvedtimestampMoved to lead_contacts. (Stays in prospeo_new_leads as the audit trail.)
rejectedNULLJam rejected, OR BC's filter auto-rejected on insert (rejected=true case). Terminal — never moves.

Why two booleans instead of one approval state?

lead_approval='approved' alone doesn't tell us whether the move has happened yet — and we can't infer it from lead_contacts because ON CONFLICT (lead_email) DO NOTHING silently drops duplicates. lead_moved_at stamps the worker's intent ("I processed this row"), regardless of whether the insert deduplicated. Distinguishing the two lets the worker run idempotently and the finalize-check be correct.

4Schema

One new table, one new column. DDL lives at the bottom of migrations.sql; applied to production Supabase via scripts/apply_scrape_requests_schema.py (idempotent — safe to re-run).

create table scrape_requests (
  id              bigserial primary key,
  -- Input fields (Jam fills these via NocoDB form)
  requested_leads integer  not null check (requested_leads between 1 and 5000),
  industries      text[]   not null default '{}',   -- empty = all 12
  skip_industries text[]   not null default '{}',
  countries       text[]   not null default '{United States,Canada}',
  notes           text,
  -- State machine (worker drives these)
  status          text not null default 'pending'
                  check (status in ('pending','running','ready','moved','rejected','failed')),
  approval        text not null default 'pending'
                  check (approval in ('pending','approved','rejected')),
  scraped_count   integer  not null default 0,
  moved_count     integer  not null default 0,
  credits_spent   numeric(10,1) not null default 0,
  created_at      timestamptz not null default now(),
  started_at      timestamptz,
  ready_at        timestamptz,
  moved_at        timestamptz,
  failed_at       timestamptz,
  email_sent_at   timestamptz,
  export_csv_path  text,
  export_xlsx_path text,
  error_message   text
);
create index scrape_requests_status_idx   on scrape_requests (status);
create index scrape_requests_approval_idx on scrape_requests (approval);

alter table prospeo_new_leads
  add column scrape_request_id bigint references scrape_requests(id);
create index prospeo_new_leads_scrape_request_id_idx
  on prospeo_new_leads (scrape_request_id) where scrape_request_id is not null;

-- Per-lead approval workflow (Flavor C).
alter table prospeo_new_leads
  add column lead_approval text
    check (lead_approval is null or
           lead_approval in ('pending', 'approved', 'rejected'));
alter table prospeo_new_leads
  add column lead_moved_at timestamptz;
create index prospeo_new_leads_pending_move_idx
  on prospeo_new_leads (scrape_request_id)
  where lead_approval = 'approved' and lead_moved_at is null;

Why prospeo_new_leads.scrape_request_id? When the move-to-lead_contacts step runs, it needs to pick exactly the rows that came from this request, not a previous one. The tag is the only reliable join key — BC's per-lead id isn't stable across our schema, and scraped_at isn't precise enough to distinguish concurrent runs.

Why the partial index? The worker's hot path is "find approved-but-not-yet-moved leads." Without the partial index, that scan would be O(all_rows_for_this_request); with it, O(approved_unmoved). With 30K+ rows and most already-moved or auto-rejected by BC, this keeps each poll cheap.

5Components

worker.py — the polling loop

Single long-running process. ~450 lines, procedural. Two responsibilities per poll cycle:

FunctionJob
sweep_stuck_running()At startup, demote any status='running' row older than 1 hour back to pending (crash recovery from OOM / redeploy / kill).
claim_pending_request()SELECT ... FOR UPDATE SKIP LOCKED LIMIT 1 on status='pending', flips to running in the same statement.
run_scrape(req)Calls bettercontact_sync.main(scrape_request_id=req.id, ...). Returns the run-summary dict (accepted / credits_spent / csv_path / xlsx_path / aborted_reason).
mark_ready(...)Writes scraped_count, credits_spent, export_csv_path, export_xlsx_path; flips to ready.
mark_failed(...)Writes error_message, flips to failed. Used by every exception path.
find_requests_with_pending_moves()Returns scrape_request ids that have any lead with lead_approval='approved' and lead_moved_at IS NULL. Cheap via the partial index.
move_approved_leads_for_request(rid)Per-request: SELECT ... FOR UPDATE SKIP LOCKED on approved-unmoved rows, INSERT INTO lead_contacts ... ON CONFLICT DO NOTHING, then UPDATE ... SET lead_moved_at=now() on the source rows. Increments scrape_requests.moved_count by the number of NEW rows inserted (dedup-skipped rows are still stamped moved_at but don't increment the counter — Jam sees only fresh contributions to the pool).
finalize_request_if_done(rid)Atomically flips status='moved' when no lead_approval='pending' rows remain AND every approved row has lead_moved_at set. Idempotent — coalesces moved_at so re-running doesn't reset the timestamp.
process_pending_lead_moves()Loops over every request with pending moves, moves each, then attempts finalize. Returns True if any work was done this cycle.
main()Endless loop: sweep once, then process_one_pending_request + process_pending_lead_moves every POLL_INTERVAL_S (default 60s). Catches transient DB errors and reconnects.

notifier.py — Resend HTTP wrapper

One file, two functions. Thin HTTP POST to https://api.resend.com/emails. Best-effort: a failure logs to stderr and returns False, never raises. The worker commits status='ready' before attempting the send, so an outage at Resend can't strand a request in a half-state.

  • send_batch_ready_email(req) — the "your batch is ready" email. HTML body shows leads scraped, credits spent, top 5 industries by count, and an "Open in NocoDB" button.
  • send_failure_email(req_id, error) — sent when the pending cycle hits InsufficientCreditsError or an unexpected exception. Subject is Lead batch #N FAILED so Jam sees it stand out.

bettercontact_sync.py — modified, not new

Two contract changes; existing CLI callers are unaffected:

  • Added scrape_request_id: int | None = None kwarg to main(), _run_category(), and _insert_leads(). When set, every row inserted into prospeo_new_leads during the run carries this tag.
  • _run_category() and main() now return a run-summary dict instead of None: {accepted, rejected, credits_spent, csv_path, xlsx_path, aborted_reason, rejected_counts}. Existing CLI callers ignore the return — no behavior change.

Infra files

  • Dockerfile.worker — python:3.12-slim, installs requirements.txt, CMD python -u worker.py. The -u matters: Railway logs only surface output when the buffer flushes; unbuffered mode shows the worker's heartbeat lines in real time.
  • railway.worker.json — service config pointing at Dockerfile.worker, restart ON_FAILURE with max 10 retries. (The existing railway.json stays untouched; it's for the refresh-cron service.)
  • LEAD_AUTOMATION.md — operator guide: env vars, NocoDB UI setup, troubleshooting, rollback.
  • LEAD_AUTOMATION_MOCKUPS.html — visual mockups used in the design review (NocoDB form, grid, expanded row, email body). Reference only.

6Key design decisions

NocoDB as the UI, not a standalone web app

Jam already lives in NocoDB to review the existing 200K pool. A new SaaS-style web app would mean another deploy target, another auth scheme, another thing to maintain. NocoDB gives us form + grid + per-row edit for free, and her bookmark is one click. Trade-off: no fancy preview / sort UI — she sees the same per-row grid she uses for lead_contacts today. We considered an Option B standalone web app in the design review (see mockups HTML) and chose NocoDB explicitly.

Polling instead of a webhook from NocoDB

NocoDB's webhook story is fragile (auth, retries, NocoDB version drift). A 60s poll is simpler, robust to Railway restarts, and the latency cost (~30s average) doesn't matter when scrapes take 1–10 minutes anyway.

Per-lead approval, not batch approval

BC's filter does most of the obvious-reject work but isn't perfect — in the E2E test it accepted "Owner, small business bookkeeper @ Crocs" (not a real Crocs employee). A batch-approve workflow would force Jam to either ship those false positives into the 200K pool or reject the whole batch over a handful of bad leads. Per-lead approval lets her cherry-pick. The downside is more clicks per batch; we mitigate by defaulting all BC-accepted leads to pending (no work needed if she approves everything) and letting NocoDB do bulk-row select for mass-approve.

Idempotent move via lead_moved_at

The worker can re-fire process_pending_lead_moves on every poll without double-moving anything: the move query filters on lead_moved_at IS NULL, and the UPDATE after the INSERT stamps moved_at regardless of whether the INSERT's ON CONFLICT skipped a row (the lead already exists in lead_contacts). Either way, the row is marked "processed" and won't appear in the next move query.

Auto-finalize over a "close batch" trigger

Jam never has to explicitly tell the worker she's done with a batch. The worker checks every cycle: if no leads are pending AND every approved lead has moved, the request flips to status='moved'. This avoids a UX dead-end where Jam approves the last lead but the batch sits in ready forever because she never clicked a "finalize" button.

Pre-charged budget cap on BC credits

BetterContact's API reports credits consumed only after a call completes. To avoid runaway spend during a slow poll, the worker reserves page_limit credits per submit (in_flight_credits) and checks spent + in_flight + page_limit ≤ max_credits before each new submit. On timeout we conservatively assume the reservation was spent (matches what BC actually charged in the post-batch1 recovery scenario).

ON CONFLICT (lead_email) DO NOTHING on the move

lead_contacts already has 230K rows. New BC leads will occasionally overlap with an existing entry; we silently skip those rather than re-insert or merge. moved_count reflects new rows only (the INSERT's rowcount), which is the number Jam cares about ("how many fresh contacts did I just unlock?"). Even when ON CONFLICT skips a row, the source row in prospeo_new_leads still gets lead_moved_at stamped so the worker doesn't keep retrying it.

Worker tags rows before commit, not after

The scrape_request_id goes onto every prospeo_new_leads row inside the existing INSERT in _insert_leads(), not in a follow-up UPDATE. If the scrape crashes mid-cycle, every row already written is correctly tagged and the next move-step will find them.

9What this plan does NOT do

  • No editing a request after submit. Jam submits a new one. (Considered; deferred until we see whether the per-lead reject workflow makes it unnecessary.)
  • No automatic load into Instantly. Leads land in lead_contacts; the existing flow takes over from there.
  • No multi-workspace sent_messages sync fix. Separate ticket; see FOLLOWUP_ANALYSIS_PLAN.md §6.
  • No multi-approver workflow. Single Jam approval is enough.
