Follow-up tracker v3 — spreadsheet shape

Replaces Jam's manual "DE Email Master Sheet → Follow Up Tracker" spreadsheet with a NocoDB-backed materialized view that has the same shape she already knows — one row per lead, paired Date + Body columns for each follow-up position — augmented with a new automated "Last reply from Instantly" column. No more manual upkeep.

Scope decision. Stakeholder review (2026-05-21) re-scoped: prioritize the spreadsheet replacement first, defer the Option D winning-reply work (Haiku-as-judge identifying the converting reply per booked lead). This doc covers the tracker shape; the winning-reply pieces ship behind a feature flag (see Phase 5 for the table + script that exist but aren't wired into the daily refresh yet).

The problem in one paragraph

Jam maintains a 1,231-row spreadsheet by hand. Every time the team sends a follow-up, she has to find the lead in the sheet, add the date + body to the next ffup column. Every time a lead replies, she has to dig into Instantly Unibox to see what they said and update the Status / NOTE. The sheet is always slightly out of date and the upkeep is a job. v3 turns the sheet into a database view that updates itself daily — and adds the lead's last reply right next to her notes so she can act on it without leaving NocoDB.

How it works

Instantly API │ ├─── /v2/emails?email_type=received ───► replies (already exists, ~17K rows) │ │ └─── /v2/emails?email_type=sent ───► sent_messages (empty today; Phase 2 fills it) │ followup_tracker_2026-05-19.csv ───► lead_outcomes (one-time CSV ingest) │ ▼ followup_tracker_mv ├─ Client, Email, Campaign, Leadlist Source ├─ Status, Qualified ├─ Initial Reply Date + Body (from replies, MIN ts) ├─ Email ffup 1 Date + Body (from sent_messages, rank=1) ├─ Email ffup 2 Date + Body (rank=2) ├─ ... up to ffup 10 (rank=10) ├─ Call ffup, NOTE (JOYCE) (from lead_outcomes CSV) └─ Last reply from Instantly ★ NEW (from replies, MAX ts) │ ▼ NocoDB

The MV pivots sent_messages from long-form (one row per outbound) into wide-form (10 ffup positions per lead) via conditional aggregates. NocoDB sees a single wide table that looks just like Jam's spreadsheet.

1.2 lead_outcomes — now mandatory in v3

In v2 this was optional. In v3 it's load-bearing — it holds the columns the API doesn't give us:

create table if not exists lead_outcomes (
  lead_email text not null,
  client text not null,
  campaign text not null default '',
  leadlist_source text,                 -- CSV "Leadlist Source"
  status_raw text,                      -- CSV "Status" verbatim (e.g. "Booked", "Asking for Proposal")
  qualified text,                       -- 'Qualified' | 'No' | 'Pending'
  note text,                            -- CSV "NOTE (JOYCE)"
  call_ffup text,                       -- CSV "Call ffup"
  source text not null default 'manual_tracker_csv',
  updated_at timestamptz default now(),
  primary key (lead_email, client, campaign)
);
create index if not exists lead_outcomes_lead_idx on lead_outcomes (lead_email);

1.3 followup_tracker_mv — the wide MV

Conditional aggregates pivot sent_messages into wide ffup-1 through ffup-10 columns. Last-reply column reads from replies.

create materialized view followup_tracker_mv as
with first_reply as (
  select distinct on (lead_email) lead_email, reply_timestamp, body
  from replies order by lead_email, reply_timestamp asc
),
last_reply as (
  select distinct on (lead_email) lead_email, reply_timestamp, body, from_address_email
  from replies order by lead_email, reply_timestamp desc
),
ranked_outbounds as (
  select s.lead_email, s.sent_timestamp, s.subject, s.body,
    row_number() over (partition by s.lead_email order by s.sent_timestamp asc) as ffup_n
  from sent_messages s
  where s.send_kind = 'unibox_manual'
)
select
  lo.client                              as "Client",
  lo.lead_email                          as "Email Address",
  lo.campaign                            as "Campaign",
  lo.leadlist_source                     as "Leadlist Source",
  coalesce(lo.status_raw, l.auto_status) as "Status",
  lo.qualified                           as "Qualified",
  fr.reply_timestamp                     as "Initial Reply Date",
  fr.body                                as "What was their initial reply",
  -- ffup 1
  max(case when ro.ffup_n = 1 then ro.sent_timestamp end)  as "Email ffup 1 Date",
  max(case when ro.ffup_n = 1 then ro.body end)            as "Email FF 1 what we sent",
  -- ffup 2 through ffup 10 (same pattern; see PLAN.v3.md §1.3 for full DDL)
  ...
  lo.call_ffup                           as "Call ffup",
  lo.note                                as "NOTE (JOYCE)",
  -- ★ NEW column
  lr.reply_timestamp                     as "Last Reply At",
  case
    when lr.from_address_email like '%@google.com'
         or lr.from_address_email like '%calendly%'
         or lr.from_address_email like '%calendar%'
    then '[Calendar invite] ' || left(lr.body, 300)
    else left(lr.body, 500)
  end                                    as "Last reply from Instantly"
from lead_outcomes lo
left join leads l                  on l.lead_email = lo.lead_email
left join first_reply fr           on fr.lead_email = lo.lead_email
left join last_reply lr            on lr.lead_email = lo.lead_email
left join ranked_outbounds ro      on ro.lead_email = lo.lead_email
group by lo.client, lo.lead_email, lo.campaign, lo.leadlist_source, lo.status_raw,
         l.auto_status, lo.qualified, fr.reply_timestamp, fr.body,
         lo.call_ffup, lo.note, lr.reply_timestamp, lr.body, lr.from_address_email;

create unique index followup_tracker_mv_pk
  on followup_tracker_mv ("Email Address", "Client", "Campaign");

Full DDL with all 10 ffup pairs in FOLLOWUP_ANALYSIS_PLAN.md §1.3.

Why a wide pivot rather than a thin "one row per (lead, ffup_n)" MV: Jam's tracker shape is what NocoDB renders by default. A thin MV would push the pivot to client-side and break the familiar layout. The conditional-aggregate approach reproduces the spreadsheet exactly. Refresh cost is OK while sent_messages is small; revisit if it grows past ~500K rows.

Known limitations

1. v3 doesn't answer "which reply converted"

By design — that's deferred to v4 (Option D). v3 ships the spreadsheet replacement first. The conversion analysis is a separate question that can be layered on later without touching v3's MV.

2. Leads with >10 manual outbounds get truncated

The MV surfaces the first 10 chronologically. Same constraint as Jam's spreadsheet. Workaround if it bites: add ffup 11–15 columns, or add a "+N more" overflow indicator.

3. "Last reply from Instantly" can be a Calendar invite

When a lead books via Calendly without typing a "yes" email, the last inbound is the calendar invite. v3 tags these with a [Calendar invite] prefix in the new column so Jam knows it's not a real reply. If she wants both the last inbound AND the last human reply visible, split into two columns (~15 min change).

4. First-backfill volume is unverified

Probe v2 timed out at 30s on 90 days. Phase 2 starts with a 7-day smoke before extrapolating.

5. step IS NULL ⟺ ue_type=3 is empirical, not guaranteed

If Instantly ever attaches step to a manual reply, those rows would miscategorize as campaign_auto and disappear from ffup columns. Re-verify quarterly.

6. Wide-MV refresh cost grows with sent_messages

Conditional aggregates over a growing table get slower. Fine at current volumes (~0 rows, scaling to ~90K after backfill). Watch performance. If refresh exceeds ~30s, switch to a thin MV (one row per (lead, ffup_n)) and let NocoDB do client-side pivot.

Things to watch out for

  • Instantly's /v2/emails does not expose in_reply_to_id (Phase 0.6).
  • Right API filter is ?lead=<email>, NOT ?lead_email= — the latter is silently ignored.
  • Existing sent_messages is empty. No data-loss risk on schema additions.
  • Calendar invite emails appear as inbound with <calendar-...@google.com> Message-ID headers. MV tags them with [Calendar invite].
  • NocoDB schema cache: any DDL change to followup_tracker_mv requires meta-sync.
  • The MV needs a unique index on ("Email Address", "Client", "Campaign") for refresh concurrently to work.
  • Bodies are HTML. Reuse html_to_text() from instantly_sync.py:293.
  • Conditional-aggregate MVs can be slow to refresh once sent_messages grows large. Watch performance; switch to thin MV + client-side pivot if it gets bad.
  • Stop maintaining the CSV after Phase 3 ships. One-time historical snapshot, not a living document.
