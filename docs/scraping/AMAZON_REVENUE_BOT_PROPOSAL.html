Amazon Revenue QA Bot

What it needs to do, the options we researched (with trade-offs), and the recommended path — with reasons. Prepared 2026-06-30.

1. Why we're building this

Right now ~50–70% of the leads we scrape are below our ideal customer — small brands that won't convert. Before we spend money enriching and emailing them, we want a quality gate that keeps only brands that are (a) actually selling on Amazon and (b) making at least $500,000/year there. Think of it as the Amazon version of the e-commerce checker we already run.

2. Requirements — what the feature must do

  • Sit inside the existing flow, after the e-commerce check and before we grab emails (so we don't pay to enrich bad leads):
    Scrape → E-commerce QA → Amazon Revenue QA (this) → Grab email → MillionVerifier → Jam approves → 200k DB
  • Check Amazon presence — is this brand actually selling on Amazon?
  • Get the brand's Amazon revenue and keep only those ≥ $500k/year (the number must be easy to change).
  • Match the company to the right Amazon brand accurately — e.g. "Blue Apple Co." must NOT be mistaken for the giant brand "Apple".
  • Output the revenue as its own column in the results.
  • Never silently drop a lead — if revenue is unknown, send it to Jam for review.
  • Be reliable, low-maintenance, and safe — no account bans or terms-of-service violations.

4. Revenue source — options, pros & cons

Option A — SmartScout, manual refresh Recommended (now)

We already have SmartScout data loaded. Periodically re-export from SmartScout and re-upload (run.py upload-smartscout).

  • ✔ Already integrated; gives brand revenue directly (no extra assembly).
  • ✔ Uses our existing plan — near-zero extra cost.
  • ✔ Legitimate — no terms-of-service / ban risk.
  • ✔ Accurate enough for a coarse $500k line (~80–90% vendor-claimed).
  • ✘ Our copy is a static snapshot (currently ~May 8) — goes stale unless we re-upload.
  • ✘ Low coverage of our leads: only ~14–20% name-match (measured — see the Coverage box below); fuller data barely helps. Refresh is manual.

Option B — SmartScout API Recommended (if we want it automated)

SmartScout's paid API returns brand revenue on demand.

  • ✔ Fresh, automated, direct brand revenue — solves the staleness problem cleanly.
  • ✔ Legitimate API — no ToS/ban risk; data updated daily.
  • ✘ Enterprise add-on, ~$399+/mo, custom pricing (must contact sales).

Option C — Keepa API Cheap fallback

  • ✔ Cheap (~$54/mo); legitimate API; can filter by brand; uses Amazon's real "bought past month" number.
  • ✘ No brand revenue directly — it's per-product, so we'd build the "list the brand's products → sum revenue" logic ourselves.
  • ✘ The sold-count is missing for low-volume products.

Option D — Rainforest API Weaker fit

  • ✔ Real-time public Amazon data; legitimate.
  • ✘ Returns units only, not revenue; no brand aggregation; pricier entry (~$83+/mo).

Option E — Helium 10 via browser automation Avoid

Log into Helium 10 on a VM and scrape its tools with Playwright.

  • ✔ Fresh, broad coverage.
  • ✘ Directly violates Helium 10's Terms of Service (no automated access) → account can be suspended without notice.
  • ✘ Google blocks automated logins; needs periodic manual re-login — not truly unattended.
  • ✘ Revenue accuracy for gating is unverified; fragile + high-maintenance (the team already failed at Amazon scraping before).

5. Amazon presence check — options

Most covered brands are confirmed "on Amazon" simply by matching SmartScout. For the rest:

  • Recommended Managed Amazon-data API (e.g. Rainforest) — reliable, ~$25–75/mo, they handle proxies/CAPTCHAs. ✔ low-maintenance
  • Avoid DIY Playwright + proxies + CAPTCHA solver — ~$100–290/mo plus heavy maintenance, fragile; Amazon's CAPTCHAs already defeated the team once.

6. Recommendation & reasons

✅ Recommended path (realistic — a filter + review queue, not a hands-off gate)

  • Brand matching: keep the strict matcher (built) + add space-insensitive normalization to recover name-variant brands. — loose matching produced $2.9B false positives; better normalization lifted coverage 14%→18%.
  • Revenue for the ~20% clean Amazon brands: gate on SmartScout (manual re-upload now; API later). — only safe, direct brand-revenue source; no ban risk; accurate enough for a coarse line.
  • Amazon presence — the higher-leverage next piece: a managed API to sort the other ~80% into "on Amazon → fetch revenue" vs "not on Amazon → out of ICP / drop." — for most leads, "are they even on Amazon?" is more decisive than the exact revenue; a managed API avoids the CAPTCHA war the team lost before.
  • Live revenue for the "on-Amazon-but-unmatched" slice: only if needed, Keepa (~$54/mo, legitimate API).
  • Unknown → Jam review, never a silent drop. — a review tail is unavoidable; the goal is to shrink it, not eliminate it.

⚠️ What we explicitly ruled out — and why

Helium 10 browser automation. It's a Terms-of-Service violation (ban risk), Google blocks the automated login (so it needs constant manual re-logins — not unattended), and its gate-accuracy is unverified. For a coarse $500k threshold the payoff doesn't justify the legal + reliability risk — especially when SmartScout already gives us brand revenue safely.

⚠️ Coverage — MEASURED, and it's the crux (not assumed)

We tested this against a 1,500-company sample of our real leads:
Match approachCoverage of our leads
Exact match vs our 275k snapshot13%
Exact match vs the fuller 513k SmartScout file14%
+ better (space-insensitive) matching18%
+ fuzzy (estimated ceiling)~20–25%
  • More/fresher SmartScout data does NOT fix this — going from 275k → 513k moved coverage 13% → 14%. Completeness was never the bottleneck; SmartScout tops out at ~20% of our leads.
  • The ~80% that don't match split roughly in half: real Amazon brands whose names just differ (Nano Bebe→"nanobebe", Linjer, Rainier Arms — recoverable only via a live per-brand lookup), and resellers / agencies / multi-brand sellers / services (Digital Sports Mgmt, Best Priced Products, Ohio Power Tool — not a single Amazon brand, largely out of ICP).
  • Implication: there is no cheap, safe, high-coverage revenue auto-gate — matching arbitrary company names to Amazon brand revenue is a hard entity-resolution problem (which is why past attempts struggled). This feature is realistically a high-confidence filter + a review queue, not a 100% auto-gate.
  • The low match rate is also a lead-sourcing signal: a large share of scraped leads aren't identifiable Amazon consumer brands in the first place.

In one sentence

This is a filter + review queue: SmartScout auto-decides the ~20% of leads that are clean, matchable Amazon brands (safely, no ban risk); a managed presence API sorts the rest into on-Amazon vs out-of-ICP; anything uncertain goes to Jam — with Helium 10 UI automation avoided entirely. A revenue number for every lead isn't achievable cheaply or safely, and the low match rate itself flags that much of the lead pool isn't Amazon-brand ICP.

Final validated design

Per lead, a cheap cascade → KEEP / DROP / REVIEW:
1 strict brand-match → in SmartScout? → yes: revenue ≥ $500k → KEEP, else DROP (~20% of leads, free)
2 else → Rainforest search (1 credit): <2 branded listings → DROP (not on Amazon) · "bought past month"×price ≥ $500k/yr → KEEP · established (has sales label or ≥1,000 ratings) but below → REVIEW · otherwise DROP (small)
3 anything unknown/low-signal → Jam review (never a silent drop)
Why it's sound: the KEEP is a revenue floor from Amazon's own "bought past month" counts — we only under-read, so a KEEP is never a false positive. DROP = not-on-Amazon or demonstrably tiny. REVIEW = a small, bounded human queue.
Provider-agnostic: the scorer only needs {title, price, bought-past-month, ratings} per result → Rainforest is proven, but the fetch layer swaps to DataForSEO/Oxylabs without touching the logic.

✅ Validated on real data (before the free trial lapsed)

  • Strict matcher kills the false positives ("Blue Apple Co." ≠ brand "Apple").
  • Revenue floor works: Nano Bebe → $2.36M/yr → KEEP (from 25 labeled products), no Helium 10.
  • Coverage measured: SmartScout name-matches ~20% of leads; of the rest, ≥14% are on Amazon (recoverable) and ~77% are not on Amazon (out of ICP — the "garbage leads" the bot is meant to filter).
  • Presence check cleanly separates real brands (nanobebe) from non-brands (Digital Sports Mgmt).
Built + local: amazon_brand_match.py (strict matcher), amazon_revenue_qa.py (cascade + $500k gate + cache), amazon_presence.py (presence + floor scorer), brand_revenue_cache table.

⚠️ Outcome / blocker (2026-07-01)

The Rainforest free trial is exhausted — they allow one free trial per organisation and suspended a second account. The method is proven; running it at scale now needs paid credits. This is a small spend decision, not a technical blocker:
  • Rainforest paid ~$83/mo — cleanest (integration already built + proven).
  • DataForSEO pay-as-you-go (~cents/request, $50 min) — likely cheaper; verify it returns "bought past month" first, then swap the fetch layer.

Implementation plan

  • Phase 0 — build (DONE, local): strict matcher, cascade, $500k gate, cache table, SmartScout provider, Rainforest floor-scorer — all built and validated on real data.
  • Phase 1 — fund a provider: pick Rainforest (paid) or verify+choose DataForSEO; put the API key in the worker's env. (This is the one gating decision.)
  • Phase 2 — wire into the pipeline: run the cascade in the worker after the e-commerce QA (brand_verify) and before email-grab; write revenue + verdict as columns on prospeo_new_leads; cache results in brand_revenue_cache.
  • Phase 3 — surface to Jam: show KEEP/DROP/REVIEW on the /batches review; the REVIEW queue is handled by a human (optionally Claude-Chrome + your Helium 10 login for spot revenue).
  • Phase 4 — schedule, tune, monitor: periodic SmartScout re-export/upload for freshness; calibrate thresholds (ratings floor, grey band) on a labeled sample; MillionVerifier + Jam approval → 200k DB (existing path).
  • Ongoing: keep the fetch layer swappable; revisit Keepa / Helium 10 only if a gap demands it.
