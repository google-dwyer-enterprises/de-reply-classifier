What the bot checks — and how it was verified

One Rainforest search call (1 credit) performs BOTH checks at once:

  • On-Amazon check — count listings whose brand/title match the company (strict + space-insensitive). 0 branded listings → DROP (not an Amazon brand).
  • $500k revenue check — a revenue floor: Amazon's own "bought in past month" labels × price × 12 across branded listings. Floor ≥ $500k → KEEP. It under-counts by construction (only labeled listings count), so a KEEP is never a false positive.
  • Established brand (≥2 listings, ≥1,000 cumulative ratings) but floor below the line → REVIEW (a human decides; the floor may just be under-reading). Small presence → DROP. API failure → REVIEW, never a silent verdict.

Key insight — the cost hierarchy is inverted vs intuition. A Rainforest check costs ~8× a BetterContact email. The meeting's "check Amazon before grabbing the email" framing assumed the email is the expensive step to protect — it's actually the cheapest paid step, and it's sunk at search time anyway. The thing to protect is the Rainforest call itself: run it after every free/cheaper rejector (free filters → Haiku gate → brand_verify), per unique company (the 3-contact cap means 1 credit covers up to 3 leads), SmartScout-first, 90-day cached.

Quality note — order-invariance: a lead is accepted only if it passes every gate, so reordering gates changes cost, never outcomes — provided no gate is skipped and Amazon REVIEWs pass through stamped rather than auto-rejected (they do, mirroring bv3's "unknowns are never auto-rejected" rule). The single untested piece is live in-worker behavior — covered by shadow mode: verdicts recorded on a real batch, compared against Jam's judgment, before the bot gets drop authority.

  • Do NOT place before brand_verify — Rainforest would bill $0.17/company for leads bv rejects for cents (resellers, MLM, no DTC store).
  • Free bonus signal: bv3 already stores an amazon_presence hint (site links to an Amazon storefront) — usable to strengthen weird-name matching at zero cost.
  • A two-phase BC flow (search enrich-free → re-enrich survivors) would honor "before email-grab" literally, but saves only ~$0.02/lead while adding real complexity — not recommended.
