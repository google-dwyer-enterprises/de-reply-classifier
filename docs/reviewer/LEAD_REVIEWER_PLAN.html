Lead Reviewer — Static Page Plan

Replace Jam's NocoDB interactions with a purpose-built Flask page. NocoDB stays read-only for audit / 200K pool. Design only — sign off here before any code lands.

Goal

Give Jam a CRM-style review UI for batches — true select all / deselect all / range select / non-consecutive multi-select with a real Apply button — that doesn't depend on NocoDB's filter-based "hiding" or its quirky bulk-edit confirmation modal.

Tech stack (all current-stack)

  • Backend: Flask (no FastAPI, no Django) — minimal, single file feasible
  • Templates: Jinja2 (Flask default)
  • Frontend: Vanilla JS for selection state + AJAX. No React, no build step.
  • CSS: Single hand-written stylesheet, ~200 lines. Same color palette as NocoDB chips.
  • DB: psycopg2 (reuse db.py)
  • Deploy: New Railway service lead-reviewer in perfect-learning, separate Dockerfile.web, runs gunicorn app:app.

UX details for the review page

  • Sticky action bar: stays at top as Jam scrolls. Apply button disabled until at least one row is selected.
  • Selection state in JS: Set of row ids; updates on checkbox toggle, shift-click, header-checkbox.
  • "Select pending only" button: in case she lands on a re-opened batch with mixed states.
  • Apply behavior: POST to /batch/<token>/bulk-update with {lead_ids: [...], status: "approved"}. Backend updates DB. Page refreshes that section (selected rows show new chip color, then fade out if filter excludes them).
  • Toast confirmation: "3 leads approved. Worker will move them within 30 seconds."
  • Mass-approve / mass-reject buttons: Same effect as setting scrape_requests.approval='approved' — the existing worker logic handles the rest.
  • Empty state: when all leads have a decision, show "Batch closed. All decisions recorded."

Auth model

  • HTTP Basic on /submit and /batches — one shared username/password (env vars), Jam enters once per device, browser remembers.
  • Per-batch token on /batch/<token> — random UUID stored in scrape_requests.review_token, embedded in the email. Same security model as NocoDB share URLs. Anyone with the token can review that batch; nobody else.
  • No login UI to build. No session management. No CSRF complications.

5DB changes

One small addition; everything else stays as-is.

alter table scrape_requests
  add column if not exists review_token uuid default gen_random_uuid();
create unique index if not exists scrape_requests_review_token_idx
  on scrape_requests (review_token);

Generated at insert time. UUID is unguessable. Used as the URL token for /batch/<token>. Replaces review_view_id / review_share_uuid / review_url (those get removed in Phase 5).

8What this plan does NOT do

  • Doesn't replace the Railway worker. It stays unchanged.
  • Doesn't replace NocoDB for audit / browsing the 200K pool. NocoDB stays as a read-only viewer.
  • Doesn't change the database schema beyond adding review_token.
  • Doesn't change BetterContact / Resend integrations.
  • Doesn't add a real login system — token + HTTP Basic is the security model.
